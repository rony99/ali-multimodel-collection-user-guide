# 用户操作步骤

> 大白话操作手册（**主线**）。门槛细则对照：[甲方要求说明.md](./甲方要求说明.md)。  
> <span style="color:#d93025">下文红色为交卷硬要求。</span>  
> <span style="color:#d93025">**正式交卷必须两块都交、平级放置**：</span>  
> ① **甲方要求的最终数据包**（对照 [样例包](./甲方数据包参考样例/20260617_gateway-raw-http/)）；  
> ② **`multi-sessions/`**（多次执行 session 校验，对照 [样例](./甲方数据包参考样例/multi-sessions/)：各模型多次独立 session + 通过情况 manifest）。  
> <span style="color:#d93025">`multi-sessions/` 里 Opus 须至少 **≥2** 次独立 session（正式合格仍按 pass@4 ≤ 60%，**建议交齐 4 次**）。只交①或只交②都不算交齐。</span>

---

## 先搞懂三件事

1. **你在出一道「给 AI 做的编程题」**  
   一份还不完整（或有 bug）的代码，让 AI 补功能或修 bug。
2. **对不对只看自动测试**  
   Docker 里跑 `tests/test.sh`：退出码 `0` = 过，否则 = 挂。<span style="color:#d93025">不要用人眼「感觉写得不错」判分。</span>
3. **三个模型都要做：千问、GLM、Opus**  
   <span style="color:#d93025">题要够难：Opus 也不能几乎每次都过；千问应更吃力。</span>数字门槛见下文第 7 步。

---

## 强烈建议：Baseline 母版 + 每次 copy 到新目录

正式用 Claude Code 做题（试难度或正式采数）前，请固定这套习惯，避免污染和泄题。

### 推荐目录习惯

```text
my-task/                          # 你的任务工作区（可自定名字）
  baseline/                       # 【母版】干净初始代码，只增不改乱
                                  # 里面可以有源码、项目内测试、工程配置
                                  # ★ 禁止放 ground_truth / 标准答案 / 答案补丁
  runs/                           # 【做题工作副本】每次从 baseline 拷一份出来
    qwen-try1/
    opus-pass1/
    opus-pass2/
    ...
  ground_truth/                   # 标准答案：单独放，与 baseline 平级
                                  # 只在你自己验题、套答案测通时用
  instruction.md                  # 题面
  ...                             # 其它交卷材料按甲方要求说明整理
```

交卷包里 Baseline 通常对应 `environment/workspace/`；上面的 `baseline/` 是你本机工作习惯，整理交卷时再拷进去即可。

### 每次让 Claude Code 做题，固定四步

```text
① 从 baseline/ 完整复制到 runs/<本次名字>/     ← 不要在母版上直接改
② 确认该副本里没有 ground_truth、答案文件     ← AI 看不到正确答案
③ 新开一个干净的 Claude Code session         ← 不要沿用旧对话
④ 在副本目录里打开 / 工作，把 instruction 发给它
```

做完：在 Docker 里对**这份副本**跑测试，记下过没过；保存本 session。  
下一轮（换模型、或同一模型再试一次）：**再 copy 一份新目录 + 再新开 session**。  
母版 `baseline/` 保持干净，方便反复复制。

### 为什么必须这样？

| 错误做法 | 后果 |
| --- | --- |
| 多个模型 / 多次尝试共用同一份改过的代码 | <span style="color:#d93025">互相污染，通过率作废</span> |
| 在唯一一份 Baseline 上直接让 AI 改 | 母版脏了，后面没法从同一起点重开 |
| 做题目录里带着 `ground_truth/` | <span style="color:#d93025">AI 可能偷看答案，题作废</span> |
| 换模型却沿用旧 session | <span style="color:#d93025">上下文串台，记录不合格</span> |

## 整条路怎么走（按顺序）

```text
① 按平台说明配好模型连接
② 准备题目材料（含干净 baseline 母版 + 单独存放的标准答案）
③ 验证：Baseline 测挂；套上标准答案测过
④ 按「copy 新目录 + 新 session」用千问试做 → 太易就加难
⑤ 同样方式用 Opus 试做满 4 次 → 不能次次过
⑥ 再看 GLM、与千问的分差
⑦ 正式采齐三模型 session
⑧ 填写 meta 等文件，整理成要求的目录结构后上交
```

## 第 1 步：按平台说明配置模型连接

让 Claude Code 用平台给你的地址和密钥访问模型。

1. 从平台领取：连接地址、密钥、如何切换模型（千问 / GLM / Opus）。  
2. 设置环境变量（名字以平台说明为准，常见是）：
   - `ANTHROPIC_BASE_URL` = 平台给的地址  
   - `ANTHROPIC_AUTH_TOKEN` = 密钥  
3. 开跑前确认：当前连的是目标模型（采千问时别误开成 Opus）。  
4. **不要**把密钥写进题目文件、代码仓库或 `task.toml`。

每次做完请保存 **session**，交卷时一并提交。

## 第 2 步：准备题目材料

至少准备这 5 样（可先按本机习惯建好，交卷时再放进规定路径）：

| 材料 | 交卷时常见路径 | 一句话 |
| --- | --- | --- |
| 初始项目（Baseline） | `environment/workspace/` | 功能未完成或 bug 还在；**不含**标准答案 |
| 题面 | `instruction.md` | 告诉 AI 要做什么；**不要写出答案** |
| 测试 | 项目内用例 + `tests/test.sh` | 自动判对错 |
| Docker | `environment/Dockerfile` | 干净可复现的运行环境 |
| 标准答案（Ground Truth） | `ground_truth/`（与 workspace **分开**） | 套上后测试必须全过 |

口诀：

```text
没套答案 → 测试必须挂
套上标准答案 → 测试必须全过
AI 按 instruction 做完 → 也用同一套测试判
```

### 2.1 推荐什么样的 `instruction.md`？（对照甲方高优）

`instruction.md` 是发给 Claude Code 的**同一份**任务说明（三个模型必须完全一样）。  
写清：要完成什么、约束是什么；**不要**泄测试断言、答案路径、标准答案内容。

甲方优先希望这类题（高优方向，出题时尽量往这边靠；细则见 [甲方要求说明.md 附录 A](./甲方要求说明.md)）：

| 高优方向 | instruction 可以怎么写 / 题怎么出 |
| --- | --- |
| **长程任务** | 用户下发任务后，模型自己连续干（assistant 执行轮次，不是用户来回聊）。**底线仍是平均 ≥ 20**；高优建议尽量做到 **100+** |
| **强遵守 CLAUDE.md / 项目规范** | 题面要求必须按仓库里的规范改；可点名遵守 `CLAUDE.md`，细节放在仓库里，不要把答案写进题面 |
| **需要 Web 搜索** | 题面只给目标，关键做法需查文档 / 资料才能选对；最终测试仍须在 Docker 里**稳定可复现**（不要绑天天变的外网） |
| **多轮用户需求** | 设计成可递进：先提一版需求，再改需求 / 推翻前面结论（采数时在**同一个 session** 里多轮说完） |

另外也鼓励（非必须）：用到 skills、MCP、subagent 拆子任务、上下文压缩、自定义工具，或要遵守代码库里**没写进题面**的隐含习惯。

**写法口诀：**

- 像真实工单：目标清晰，但别写成逐步答案  
- 约束写清楚（规范、范围、不能破坏什么）  
- 难度靠「信息分散在代码库 + 规范 + 多文件改动」，不靠题面泄题  
- 三模型始终粘贴**同一份** `instruction.md`

## 第 3 步：测试怎么写？

### 3.1 判分规则

- **通过** = `test.sh` 退出码 `0`  
- **不通过** = 非 `0`  
- 不要靠「我感觉写得挺好」

### 3.2 通常要两类东西

**A. 测试用例**（真正检查功能；可放在项目内如 `tests/acceptance/`）：

```python
def test_add_works():
    from myapp.math_utils import add
    assert add(2, 3) == 5

def test_add_handles_zero():
    from myapp.math_utils import add
    assert add(0, 0) == 0
```

写法：先想清楚「做对了长什么样」→ 写成断言 → 覆盖题面主功能 → 故意写错应测不过。

**B. 统一入口 `tests/test.sh`：**

```bash
#!/bin/bash
set -euo pipefail

SOURCE_DIR="${SOURCE_DIR:-/app/source}"
export PYTHONPATH="${SOURCE_DIR}:${PYTHONPATH:-}"

set +e
python3 -m pytest "${SOURCE_DIR}/tests/acceptance/" -q
code=$?
set -e

mkdir -p /logs/verifier 2>/dev/null || true
if [ "$code" -eq 0 ]; then
  echo 1 > /logs/verifier/reward.txt
else
  echo 0 > /logs/verifier/reward.txt
fi

exit "$code"
```

```bash
chmod +x tests/test.sh
```

### 3.3 写完先自测两遍

| 情况 | 期望 |
| --- | --- |
| 只用 Baseline 跑测试 | **必须失败** |
| 把标准答案套进副本再跑 | **必须全部成功** |

初始也能全过 → 测试太松或题太简单。  
标准答案也过不了 → 先修答案或测试。

### 3.4 禁忌

- 空壳：`test.sh` 里直接 `exit 0`  
- 只检查「文件在不在」、不检查功能  
- 依赖要登录的外网、天天变的网页、真实密钥  
- 把测试断言细节写进题面（泄题）

## 第 4 步：Dockerfile 怎么写？怎么验？

### 4.1 是什么？

一份可重复的「组装说明书」：干净环境里装依赖、放代码、跑测试。别人按同一 Dockerfile 构建，结果应一致。

### 4.2 最小例子（Python）→ `environment/Dockerfile`

```dockerfile
FROM ubuntu:24.04

ENV DEBIAN_FRONTEND=noninteractive

RUN apt-get update && apt-get install -y --no-install-recommends \
    python3 \
    python3-pip \
    && rm -rf /var/lib/apt/lists/*

RUN python3 -m pip install --no-cache-dir --break-system-packages \
    pytest==8.4.1 \
    pyyaml==6.0.2

RUN mkdir -p /app/source /logs/verifier
COPY workspace/ /app/source/

WORKDIR /app/source
# RUN python3 -m pip install --no-cache-dir --break-system-packages -e .

WORKDIR /app
```

注意：依赖**写死版本**；不要写真实密钥；代码路径与 `test.sh` 一致（上例为 `/app/source`）。

### 4.3 怎么验证？

在 `environment/` 附近：

**① 构建**

```bash
cd environment
docker build -t my-task-env .
```

建议半小时内能建好。

**② Baseline：应失败**

```bash
docker run --rm -v "$(pwd)/../tests:/tests:ro" my-task-env \
  bash /tests/test.sh
```

（也可把 `tests/` 打进镜像再跑；关键是能执行到同一套 `test.sh`。）

**③ 套标准答案：应成功**

1. **复制**一份 workspace / baseline（别改坏母版）  
2. 把 `ground_truth/` 同路径文件覆盖进**这份副本**  
3. 同一镜像、同一 `test.sh` 再跑  

```text
同一套 Docker + 同一套测试
  没答案 → 挂
  有答案 → 过
```

两步都成立，再去跑模型；否则通过率无意义。

## 第 5 步：题面与标准答案

1. **`instruction.md`**：按第 2.1 节写好（高优方向优先）；说明要完成什么、有哪些约束。  
   - **不要**写测试断言全文、答案路径、标准答案内容。  
   - 三个模型永远用**同一份**。  
2. **`ground_truth/`**：按与 Baseline **相同相对路径**放「已做对」的文件；与 `baseline/` / `workspace/` **分开放**。  
3. （推荐）材料齐后做 **上传前预检**：让 Agent 读取并执行 [上传前预检/SKILL.md](./上传前预检/SKILL.md)（只需提供任务包路径）。  
   - 预审 ≠ 审核通过；通过率须自测；**以实现网和甲方的后期审核为准**。

## 第 6 步：用 Claude Code 做题（正式流程）

### 铁律

1. **每次：新目录副本 + 新 session**（见文首「强烈建议」）  
2. **同一模型、同一题只用一个 session**，不要拆成两个拼  
3. **三模型 `instruction.md` / Baseline / 测试 / Docker 完全一致**  
   - 尤其 **`instruction.md` 必须同一份**（同一内容发给三个模型），不能给 Opus「更细」、给千问「更粗」  
   - Baseline、测试、Docker 也不能中途改完再对比通过率  

### 操作清单

1. `cp -R baseline runs/<模型>-<第几次>/`（或等价复制）  
2. 确认副本内**没有** `ground_truth`、答案补丁、泄题文件  
3. **新开** Claude Code session，在该副本目录工作  
4. 把 `instruction.md` 发给它  
5. 做完后，对**该副本**在 Docker 里跑 `test.sh`，记下过/挂  
6. **保存完整 session**  
   - **主会话**：一个 `.jsonl`（`<session_id>.jsonl` 或 `session.jsonl`）  
   - **有子 Agent 时**：交齐全部 subagent 的 `.jsonl`（常见：`<session_id>/subagents/`；样例里也可能是模型目录下直接 `subagents/`）  
7. 下一轮：再 copy、再新开 session（不要复用脏目录或旧对话）

交卷时：
1. 按甲方原格式整理 **任务数据包**（对照 [甲方数据包参考样例](./甲方数据包参考样例/)）；包内可有 `trajectories/`。  
2. **另外**再建平级目录 **`multi-sessions/`**，放入各模型多次独立 session（尤其 Opus ≥2，建议 4），并在该目录写 `manifest.json` 标明每次是否通过测试。  

> <span style="color:#d93025">用户交的是 Claude Code 主会话 `.jsonl`（及有则 subagent）</span>；包内 `trajectories/` 与 `multi-sessions/` 均按此组织。Gateway / Anthropic call-level 转换由平台处理，用户不必手写 call-level schema。

## 第 7 步：把难度调到合格

判定一律看测试过没过。正式数字用「同一题独立做 4 次里过了几次」（pass@4）。

### 7.1 合格长什么样？

| 要求 | 人话 |
| --- | --- |
| <span style="color:#d93025">Opus 通过率 ≤ 60%</span> | <span style="color:#d93025">Opus 做 4 次，最多过 2 次（正式门槛）</span> |
| <span style="color:#d93025">**交卷 Opus 轨迹 ≥ 2 次**</span> | <span style="color:#d93025">除甲方数据包外必须交 `multi-sessions/`；Opus 至少 2 份独立 `.jsonl`；**建议交齐 4 次**</span> |
| <span style="color:#d93025">偏序 Opus > GLM > 千问</span> | <span style="color:#d93025">甲方硬门槛（pass@4）</span> |
| <span style="color:#d93025">Opus − 千问 ≥ 20%</span> | 平台附加；如 Opus 50%、千问 25% → 差 25%，可以 |
| GLM 略高于千问 | 平台建议，夹在中间即可，无固定差分% |
| <span style="color:#d93025">平均 **assistant 执行轮次** ≥ 20</span> | <span style="color:#d93025">**底线 / 硬门槛**</span>（必须达到；模型自己干的轮次，不是用户来回聊） |
| Opus 挂了是因为题难 | 不是 Docker / 缺依赖 |

试难度时同样遵守：**每次 copy 新目录 + 新 session**；改完题或测试后，从干净 baseline 重新开跑。

### 7.1b 一次交多道题时（批次质量）

单题可以很难（Opus 也经常挂），但**一批题交上去**时：

| 要求 | 人话 |
| --- | --- |
| <span style="color:#d93025">不能「全军覆没」</span> | <span style="color:#d93025">禁止整批都是「三个模型都测不过」的题</span> |
| <span style="color:#d93025">**至少一半 Opus 能做出**</span> | <span style="color:#d93025">所交题目中，**≥ 50%** 须为：Opus 在该题的独立多次尝试里**至少有一次通过测试**</span> |

说明：

- 「Opus 可以成功解决」= 该题 Opus 并非 0 次通过（例如 pass@4 为 1/4 或 2/4 都算「能做出」）  
- 这与「Opus pass@4 ≤ 60%」不矛盾：题仍要够难，但不能整批发「谁都做不出」的题  
- 只交 1 道题时：该题仍须满足第 7.1 节单题门槛；本条主要约束**多题一并提交**

### 7.2 推荐顺序

**先千问**

| 看到什么 | 怎么办 |
| --- | --- |
| 轻松做对 | 加难 |
| 经常做不对 | 去试 Opus |

**再 Opus（凑满 4 次）**

| 看到什么 | 怎么办 |
| --- | --- |
| 过了 3～4 次 | 继续加难 |
| 大约 ≤ 2 次过 | 算与千问分差 |
| 几乎全挂且怀疑环境 | 先修 Docker / 测试 |

**最后分差与 GLM**

1. <span style="color:#d93025">**甲方硬门槛（偏序）**：pass@4 须 **Opus > GLM > 千问**</span>  
2. <span style="color:#d93025">平台附加：`Opus通过率 − 千问通过率` ≥ 20%</span>  
3. 平台建议：GLM 比千问略高即可（无固定百分比）  

### 7.3 怎么加难？

可叠加：强制遵守 `CLAUDE.md`；细节藏在代码里；需查文档（最终测试仍须 Docker 内可复现）；多文件长链路；多轮改需求；把测试写严。

### 7.4 小循环

```text
千问太容易？ → 加难 → 再试千问
千问吃力、Opus 次次过？ → 再加难 → Opus 再做 4 次
Opus 大约 ≤2/4 过？ → 看与千问是否差 ≥20%
差不够？ → 再加难 → 从干净 baseline 重测
差够了？ → 看偏序 Opus > GLM > 千问；再看 GLM 是否略高于千问
环境导致挂？ → 先修环境
```

调合格后：把各次独立 run 的 session 收进平级 **`multi-sessions/<模型>/`**（Opus **至少 2 次，建议 4 次**），在 `multi-sessions/manifest.json` 登记 session_id、模型、是否测过；再按第 8 步整理甲方原格式数据包。

## 第 8 步：填写 meta 等文件，整理成要求结构后提交

采齐三模型 session、难度调合格之后：按下面的**目录结构**把材料收拢成一道题一个文件夹，并填好 `meta.json`、`task.toml`、rubrics 等。

**字段怎么写、必填项是什么**：不要凭感觉编，请对照：

| 参考 | 用途 |
| --- | --- |
| [甲方要求说明.md](./甲方要求说明.md) 第 5 节 + 附录 | meta / rubric / task.toml 怎么写、交卷 Checklist |
| [甲方数据包参考样例/](./甲方数据包参考样例/) | 看真实目录；`模板/` 可抄 meta、rubric |

### 8.1 要求的目录结构

目录名只用英文、数字、`_`、`-`、`.`。<span style="color:#d93025">交卷时交**两块平级**内容：</span>

| 目录 | 含义 |
| --- | --- |
| `<task_id>/` | <span style="color:#d93025">**甲方原格式数据包**</span>（结构对照 [样例包](./甲方数据包参考样例/20260617_gateway-raw-http/)） |
| `multi-sessions/` | <span style="color:#d93025">**必交附加**：三模型多次独立 session + 通过情况 `manifest.json`</span> |

```text
<submit_root>/
  <task_id>/                         # ① 甲方原格式（勿改成乱目录）
    instruction.md
    task.toml
    meta.json
    manifest.json                    # 包内采集说明
    README.md
    environment/
      Dockerfile
      workspace/
      workspace.tar.gz
    ground_truth/
    solution/solve.sh                # 可选
    tests/
    rubrics/
    trajectories/                    # 甲方结构中的轨迹
      claude-opus-4-8/
      qwen-3.7-max/
      glm-5.2/
    reports/
    scores/

  multi-sessions/                    # ② 与数据包平级（必须另交）
    claude-opus-4-8/
      <session_id>.jsonl             # ≥2，建议 4
    qwen-3.7-max/
    glm-5.2/
    manifest.json                    # session_id ↔ 模型 ↔ eval_pass
```

示意目录：[甲方数据包参考样例/multi-sessions/](./甲方数据包参考样例/multi-sessions/)。

### 8.2 整理时建议顺序

1. 按甲方原格式整理 `<task_id>/` 数据包（对照样例）。  
2. 建平级 **`multi-sessions/`**，放入各模型多次 session。  
   - 千问 / GLM：至少各有完整轨迹；为算分差建议多次。  
   - **Opus：≥2 次**（核验通过率）；**建议 4 次**对齐 pass@4。  
3. 写好 **`multi-sessions/manifest.json`**（每条含 `session_id`、`model_id`、`eval_pass`）。  
4. 补齐包内 `reports/`、`scores/`（若平台另有打分可后补）。

## 常见踩坑

| 坑 | 为什么不行 |
| --- | --- |
| 在母版 Baseline 上直接让 AI 改 | 无法从同一起点重开；请 copy 后再做 |
| 做题目录里带着标准答案 | AI 可能偷看；GT 与 baseline 分开放 |
| 三个模型题面不一致 | <span style="color:#d93025">**`instruction.md` 必须同一份**</span>；不能给 Opus 更细的说明 |
| 三个模型共用同一份脏代码 | 互相影响，成绩不准 |
| 换模型却不新开 session | 上下文串台 |
| 目录乱、meta 漏填 | 对照甲方要求说明第 5 节与甲方数据包参考样例补齐 |
| 只交一个大文件夹、没有 multi-sessions | <span style="color:#d93025">须另交平级 `multi-sessions/`（多次 session + 通过情况）</span> |
| Opus 只跑 1 次就说「够难」 | <span style="color:#d93025">看同一题 **4 次**；`multi-sessions` 至少先交 **≥2** 次 Opus 轨迹</span> |
| 只交甲方包不交 multi-sessions | <span style="color:#d93025">无法核验 Opus 通过率</span> |
| 其实是 Docker 坏了 | 先修环境再算通过率 |
| 一批题全是「谁都做不出」 | 多题提交时 ≥ 50% 须 Opus 至少能过一次 |
| 只交题目、不交 session | 必须交主会话 `.jsonl`；有子 Agent 时同名文件夹一并交 |
| 不按平台配置连接 | 按平台地址 / 密钥配置后再采 |
| `test.sh` 永远 `exit 0` | 空壳测试，会被打回 |
