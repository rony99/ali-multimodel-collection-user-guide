## 1. 评测定位与目标

| 项 | 内容 |
| --- | --- |
| 定位 | Agentic coding（编程 / 调试 / Shell）多模型对比评测 |
| 目标岗位 | 理工类程序员（前后端、数据分析、测试、DevOps 等） |
| 对比模型 | `claude-opus-4.8`、`glm-5.2`、`qwen-3.7-max` |
| 评分 | `tests/test.sh`（exec）+ Rubric 多维度打分 |
| 通过率 | **pass@4**，仅统计 unit-tests（`test.sh`）结果 |
| 需求关系 | **需求 0**（自主出题）与 **需求 1**（高优维度）独立 |

核心原则：题目宜难；必须可自动判分；三模型成绩须拉开，以发现相对薄弱点。

## 2. 题目准入与禁止事项

### 2.1 宜收录

- 真实工程场景（前后端、测试、运维、数据分析等）
- 新功能、缺陷修复、重构、补测试 / CI 等
- 具备可执行验收测试
- 可提供标准答案，且套用后测试全部通过

### 2.2 禁止

- 直接改造公开 GitHub Issue 作为题目（须私有题意）
- 依赖外部登录、强时效网页、真实密钥方可运行
- 玩具题、改动极少即可通过
- 空壳测试（如 `test.sh` 恒为 `exit 0`）

### 2.3 标签与集合上限

填写 `meta.json` 时涉及两类标识（**勿混用**）：

| 类型 | 用途 | 取值 |
| --- | --- | --- |
| Rubric 任务类型 | 评分维度权重 | 四选一：`code_generation` / `bug_fix` / `code_qa` / `refactor` |
| Taxonomy 标签 | 分类统计 | 语言、意图二级标签、应用领域等，见 [参考/Taxonomy标签.md](./参考/Taxonomy标签.md) |

按 `code_lang × task_type × application` 分组，**每组题目数 ≤ 100**。标签须如实；不必铺满笛卡尔积。

## 3. 单题必备材料

### 3.1 核心五件

| # | 材料 | 路径 | 要求 |
| --- | --- | --- | --- |
| 1 | Baseline | `environment/workspace/` | 完整初始代码库（见 3.2） |
| 2 | 题面 | `instruction.md` | 发给 Claude Code 的任务说明（见 3.3） |
| 3 | 测试 | 项目内用例 + `tests/test.sh` | 仅以测试执行结果判定对错 |
| 4 | Docker | `environment/Dockerfile` | 可复现构建；建议 **30 分钟内**完成 |
| 5 | 标准答案 | `ground_truth/` 和/或 `solution/solve.sh` | 见 3.4 |

验收关系：

```text
未套标准答案 → test.sh 必须失败
套用标准答案 → test.sh 必须全部通过
按 instruction 完成后 → 使用同一套 test.sh 判定
```

另须交付：rubrics、`meta.json`、`task.toml`、三模型 session、reports / scores 等（第 5 节）。

### 3.2 Baseline（`environment/workspace/`）

须为 Agent 动手前的**完整项目**，至少包括：

- 业务 / 源码
- 用于验收功能的测试用例
- 工程配置（如 `package.json`、`pyproject.toml`、`go.mod`、`CLAUDE.md` 等）

此时题面功能未完成或缺陷仍在 → 运行测试应失败。

**不得**纳入（与 `.gitignore` 一致的思路）：

- `node_modules/`、`__pycache__/`、`.venv/`、`dist/`、`build/`、`.next/` 等
- `.git/` 完整历史（一般勿打包）
- IDE 缓存、日志、临时文件
- 真实密钥、含秘密的 `.env`

依赖须在 Dockerfile 中按**固定版本**安装。

### 3.3 题面（`instruction.md`）

发给 Claude Code、在 Baseline 上新增功能或修复缺陷的说明。**三个模型必须使用同一份内容。**

应写明：目标、完成标准、约束（如须遵守 `CLAUDE.md`）。

不得写入：测试断言原文、隐藏样例、`ground_truth` 路径或答案内容、真实密钥。

高优题型建议见 [附录 A](#附录-a-需求-1高优任务类型)；写法参见 [用户操作步骤.md §2.1](./用户操作步骤.md)。

### 3.4 标准答案（Ground Truth）

须与 `instruction.md` 配套：按题面完整实现后应能通过全部测试。做题过程中 Agent **不得**访问答案。

| 形态 | 说明 |
| --- | --- |
| `ground_truth/` 文件树 | 仅含相对 Baseline 变更 / 新增文件；相对路径与 `workspace/` 一致；覆盖后 `test.sh` 全过 |
| `ground_truth/ground_truth.patch` | 统一 diff；适用于删除文件等不便「拷贝覆盖」表达的情况 |
| `solution/solve.sh` | 一键把 workspace 套成答案态再跑 `test.sh`。**已有可用的 `ground_truth/` 文件树时非必须**；有则可方便验收复现 |

示例：

```text
environment/workspace/src/api.py   # Baseline
ground_truth/src/api.py            # 答案（同相对路径）
ground_truth/README.md             # 套用说明
```

`solve.sh` 思路示例：

```bash
#!/bin/bash
set -euo pipefail
cp -R ground_truth/. environment/workspace/
# 或：cd environment/workspace && patch -p1 < ../../ground_truth/ground_truth.patch
```

## 4. 合格门槛

判定方式：在 Docker 中执行 `tests/test.sh`。退出码 `0` = 通过，非 `0` = 不通过。

> 下文 <span style="color:#d93025">红色</span> 为交卷硬要求 / 易踩坑要点。

### 4.1 甲方硬性要求

| 项目 | 要求 |
| --- | --- |
| 模型与 session | <span style="color:#d93025">三模型均须有完整 Claude Code session</span> |
| 平均执行轮次 | <span style="color:#d93025">assistant **执行轮次**平均 **≥ 20**（**底线**；非用户来回对话次数）</span> |
| Opus 通过率 | <span style="color:#d93025">同一题 Opus **独立 4 次**，pass@4 **≤ 60%**（至多通过 2 次）</span> |
| 区分度偏序 | <span style="color:#d93025">pass@4：**claude-opus-4.8 > glm-5.2 > qwen-3.7-max**</span>（三模型均按同题独立多次统计；交卷轨迹见 4.2） |
| 失败归因 | Opus 失败须因题目难度，而非环境故障 |
| Docker | 可构建（**30 分钟内**成功为宜）；依赖钉版本；无真实凭证 |
| Baseline / GT | <span style="color:#d93025">初始测挂；套标准答案后测过</span> |
| Rubric | 5 分制，**3** 为 borderline；correctness 不得与 `test.sh` 冲突；须有 agreement score |
| Session | 主会话必交；有委派则 subagent 一并交齐 |

**pass@4**：同一题、同一测试口径下，某模型干净重做 4 次，统计测试通过次数。通过 2 次 = 50%。不是 4 道不同题目。

### 4.2 本平台交卷附加要求

| 项目 | 要求 |
| --- | --- |
| 交卷组成 | <span style="color:#d93025">**甲方原格式数据包** + 平级 **`multi-sessions/`**</span>（多次 session + 目录内 manifest） |
| Opus 核验轨迹 | <span style="color:#d93025">除甲方数据包外，**必须提交 `multi-sessions/`**；其中 Opus 至少 **≥2** 次独立完整做题轨迹</span>。正式门槛仍为 pass@4（**建议交齐 4 次**） |
| Opus − 千问 | <span style="color:#d93025">`pass@4(opus) − pass@4(qwen) ≥ 20%`</span> |
| GLM vs 千问 | GLM **略高于**千问即可（无固定百分比） |
| 多题批次 | <span style="color:#d93025">一次提交多道题时：不得整批均为「三模型均测不过」；其中 **≥ 50%** 的题须为 Opus **至少有一次**通过测试</span>（与单题 Opus pass@4 ≤ 60% 并存） |

> 甲方检查清单 v1.1 以偏序为主；上表分差、批次比例与「Opus ≥2 次核验轨迹」为本平台交卷要求。

### 4.3 采集纪律

1. **每模型、每题：新开干净 session，且整题仅使用该 session**  
   - 禁止沿用其他任务残留对话  
   - 禁止将同一题拆成多个 session 拼凑  
   - 有 subagent 时，主 / 子会话属同一会话体系，须一并提交  
   - 建议：Baseline 母版 + 每次 copy 至新目录后开跑；做题目录不得含 `ground_truth`（见操作步骤）

2. **三模型条件完全一致**  
   - <span style="color:#d93025">**`instruction.md` 必须同一份**</span>  
   - Baseline、测试、Docker、判分口径不得因模型而异  
   - 禁止中途修改测试或 Dockerfile 后再对比通过率

## 5. 交付物结构与填写规范

交付两类内容：

1. **项目相关**：题面、代码、测试、Docker、标准答案、rubrics、meta、`task.toml` 等  
2. **Session**：各模型主会话 `.jsonl`；有委派时含 subagent 日志  

### 5.1 目录结构

目录名仅允许**字母、数字、下划线、短横线、点号**（即英文数字与 `._-`，不含中文、空格、shell 特殊字符）。`task_id` 须与**甲方数据包目录名**一致。

交卷时提交**两块平级**内容：

| 目录 | 含义 |
| --- | --- |
| `<task_id>/` | **甲方原格式数据包**（对照 [甲方数据包参考样例](./甲方数据包参考样例/)） |
| `multi-sessions/` | **附加**：三模型多次独立 session + 目录内 `manifest.json` |

```text
<submit_root>/
  <task_id>/                       # ① 甲方原格式
    instruction.md
    task.toml                      # 禁止真实密钥 → 附录 D
    meta.json
    manifest.json
    README.md
    environment/
      Dockerfile
      workspace/                   # Baseline（不含 ground_truth）
      workspace.tar.gz
    ground_truth/
    solution/solve.sh              # 可选
    tests/
      test.sh
      test_outputs.py
    rubrics/
      global_rubric.yaml
      task_rubric.yaml
    trajectories/                  # 甲方结构中的轨迹
      claude-opus-4.8/             # 或 claude-opus-4-8
      qwen-3.7-max/
      glm-5.2/
    reports/
    scores/

  multi-sessions/                  # ② 与数据包平级（必须另交）
    claude-opus-4-8/
      <session_id>.jsonl           # Opus ≥2，建议 4
    qwen-3.7-max/
    glm-5.2/
    manifest.json                  # session_id ↔ 模型 ↔ eval_pass
```

| 路径 | 说明 |
| --- | --- |
| `<task>/instruction.md` 等 | 甲方数据包字段；见样例 |
| `multi-sessions/` | 多次独立 session，用于核验通过率 |
| `multi-sessions/manifest.json` | 各次 session 与是否测过 |

### 5.2 `meta.json`

须包含：taxonomy + 技术栈标签；一句话介绍；难度评估（标注背景 + 预估用时）；rubrics 评分的 agreement score。

| 字段（同义名可择一） | 要求 |
| --- | --- |
| `task_id` | **等于**任务目录名 |
| `rubric_task_type` / `task_type` | Rubric 四选一 |
| `labels.code_lang` | 主语言（Taxonomy） |
| `labels.application` | 应用领域（Taxonomy） |
| `labels.task_type`（推荐） | 意图二级标签，可多选数组（**非** Rubric 四选一） |
| `one_liner` / `one_sentence_summary` | 一句话介绍 |
| `difficulty` 及/或 `annotator_background` + `completion_time_min` | 难度评估 |
| `agreement_score` | 评分一致性 |
| `tags` | 技术栈 + 意图 + 高优能力等 |

可选：`model_notes`、`source_repo`、`version_window` 等。

枚举见 [参考/Taxonomy标签.md](./参考/Taxonomy标签.md)。模板：[meta.最小必填.json](./甲方数据包参考样例/模板/meta.最小必填.json)、[meta.样例.json](./甲方数据包参考样例/模板/meta.样例.json)。

```json
{
  "task_id": "your_task_dir_name",
  "rubric_task_type": "code_generation",
  "labels": {
    "code_lang": "python",
    "task_type": ["feature"],
    "application": ["Backend_Infrastructure"]
  },
  "tags": ["python", "fastapi", "feature"],
  "one_liner": "简述本题测什么、为何值得作为评测题。",
  "difficulty": "hard",
  "annotator_background": {
    "industry": "互联网",
    "years_experience": 5,
    "role": "后端工程师"
  },
  "completion_time_min": 120,
  "agreement_score": 0.85
}
```

### 5.3 Rubrics

要求：整体规范 + per-task；**5 分制**，**3** = borderline；correctness 不得与 exec 冲突；per-task 各维度 `task_specific_criteria` 为**非空数组**；指定 `task_type` 即可继承全局 1～5 定义；须有 agreement score。

| 文件 | 作用 |
| --- | --- |
| `rubrics/global_rubric.yaml` | 全局维度与分档；通常直接使用样例 |
| `rubrics/task_rubric.yaml` | 本题各维度具体要点；**每题必填** |

#### 评分维度（1～5）

| 维度 | 含义 | 5 分要点（摘要） |
| --- | --- | --- |
| correctness | 是否正确完整解决问题 | 可运行并通过全部测试（含边界） |
| code_quality | 可读性、规范、可维护性 | 结构清晰、命名规范、无异味 |
| reasoning | 分析、规划、决策 | 理解问题、方案清晰、可调整策略 |
| tool_usage | 工具使用是否合理 | 选型准确、调用有目的 |
| efficiency | 路径效率 | 步骤少、少冗余 |
| explanation | **仅 code_qa** | 准确完整并引用代码证据 |

#### 维度权重（四类任务相同）

| 维度 | 权重 |
| --- | --- |
| correctness | 0.35 |
| code_quality | 0.25 |
| reasoning | 0.15 |
| tool_usage | 0.10 |
| efficiency | 0.15 |

模板：[global_rubric.yaml](./甲方数据包参考样例/模板/global_rubric.yaml)、[task_rubric.空白模板.yaml](./甲方数据包参考样例/模板/task_rubric.空白模板.yaml)、[task_rubric.样例.yaml](./甲方数据包参考样例/模板/task_rubric.样例.yaml)。

```yaml
task_id: "your_task_dir_name"
task_type: "code_generation"

task_specific_criteria:
  correctness:
    - "acceptance 测试全部通过（reward=1）"
    - "题面要求的功能行为正确（写明本题行为）"
  code_quality:
    - "接入现有模块，不破坏无关行为"
  reasoning:
    - "正确识别关键概念并选择合理改法"
  tool_usage:
    - "阅读相关测试并以测试验证"
  efficiency:
    - "无大量无关改动或无效试错"
```

说明：测试全过时 correctness 通常为 **5**；各维度至少一条具体要点。

## 附录 A. 需求 1：高优任务类型

与需求 0 **独立**；下列为并列优先方向，**不能替代**难度 / 通过率硬门槛。长程 **100+** 执行轮次为**建议**；底线仍为平均 turns **≥ 20**。请在 `tags` 中如实标注。

### 高优

| 类型 | 说明 |
| --- | --- |
| 长程任务 | 一次任务下发后，LLM **执行轮次**尽量 **≥ 100**（建议）；底线仍 ≥ 20 |
| 强遵守 CLAUDE.md / 规范 | 须按仓库规范修改，不可仅看题面 |
| 需要 Web 搜索 | 须查文档 / 资料；最终测试须在 Docker 内稳定可复现 |
| 多轮用户需求 | 递进需求、修改或推翻先前要求 |

### 其他鼓励

skills；上下文压缩；MCP；subagent；自定义工具；遵守题面未写明的隐含规范。

## 附录 C. 提交前 Checklist

未勾选项视为未达标。

### C1. 合规

- [ ] 私有题意；未直接改造公开 GitHub Issue  
- [ ] 真实工程场景；难度充分  
- [ ] `instruction.md` 不泄测试断言 / 答案路径  
- [ ] 无真实密钥；无外部登录 / 强时效依赖  

### C2. Baseline / GT / 测试 / 环境

- [ ] `workspace/` 为完整初始库；未打包 ignore 类产物  
- [ ] GT 与 instruction 配套（文件树 / patch / `solve.sh`）  
- [ ] Baseline：`test.sh` FAIL；套 GT 后 PASS（reward=1）  
- [ ] 非空壳测试；correctness 与测试一致  
- [ ] Docker 可构建（建议 ≤30 分钟）；依赖钉版本  
- [ ] 做题时 Agent 不可见 `ground_truth/`  

### C3. 文件齐全

- [ ] 已交 **甲方原格式数据包** + 平级 **`multi-sessions/`**；目录名合法  
- [ ] `multi-sessions/manifest.json`（session_id ↔ 模型 ↔ 是否通过测试）  
- [ ] 数据包内 `instruction.md`、`task.toml`、`meta.json`、`README.md`  
- [ ] `environment/Dockerfile` + `workspace/`（建议 tar.gz）  
- [ ] `tests/test.sh`（建议 `test_outputs.py`）  
- [ ] `ground_truth/` 和/或 `solution/solve.sh`  
- [ ] `meta.json` 必填齐全  
- [ ] `rubrics/global_rubric.yaml` + `task_rubric.yaml`  
- [ ] `multi-sessions/` 三模型均有 session；**Opus ≥ 2 次**（建议 4 次）  
- [ ] `reports/` + `scores/rubric_scores.json`（建议）  
- [ ] 已运行 [上传前预检](./上传前预检/SKILL.md)（预审≠实现网/甲方后期审核）  

### C4. Session 与一致性

- [ ] 每模型每题新开 session；同题同模型不拆分拼凑  
- [ ] 三模型同一份 `instruction.md` 与同一 Baseline / 测试 / Docker  
- [ ] 主会话齐全；有委派则 subagent 交齐  

### C5. 难度与区分度

- [ ] 平均 turns ≥ 20（底线）  
- [ ] Opus pass@4 ≤ 60%  
- [ ] opus > glm > qwen  
- [ ] Opus − 千问 ≥ 20%；GLM 略高于千问  
- [ ] Opus 失败归因于题难  
- [ ] **多题一并提交时**：非整批三模型全挂；其中 ≥ 50% 的题 Opus 至少有一次测过  

### C6. 高优（如适用）

- [ ] （建议）长程执行轮次 100+；及/或 CLAUDE.md / Web 搜索 / 多轮需求等  

### C7. 禁止行为

- [ ] 未伪造 session  
- [ ] 未通过更换测试口径凑通过率  
- [ ] 环境未通时未强行充数

## 附录 D. `task.toml` 配置说明

任务级配置。**禁止**写入真实 token / API Key。

### 作用

约定 verifier / agent / 镜像构建超时与资源提示；元数据可与 `meta.json` 对齐。密钥仅可置于本机环境变量或平台下发的连接配置。

### 示例

```toml
version = "1.0"

[metadata]
author_name = "your_name"
author_email = "you@example.com"
difficulty = "hard"
category = "software-engineering"
code_lang = "python"
rubric_task_type = "code_generation"
label_task_types = ["feature"]
application = ["Backend_Infrastructure"]

[verifier]
timeout_sec = 300.0

[agent]
timeout_sec = 3600.0

[environment]
build_timeout_sec = 600.0
cpus = 1
memory = "2G"
storage = "10G"
```

### 字段说明

| 段 / 字段 | 含义 |
| --- | --- |
| `[metadata]` | 作者、难度、语言、Rubric 类型、意图标签、领域等 |
| `rubric_task_type` | 四选一 |
| `label_task_types` | 意图标签列表 |
| `application` | 领域列表 |
| `[verifier].timeout_sec` | `test.sh` 最长等待（秒） |
| `[agent].timeout_sec` | Agent 做题最长等待（秒） |
| `[environment].build_timeout_sec` | 镜像构建最长等待（秒） |
| `cpus` / `memory` / `storage` | 资源提示 |

### 禁止示例

```toml
# 禁止
api_key = "sk-xxxx"
token = "真实密钥"
```
